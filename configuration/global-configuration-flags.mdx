This page documents the **runtime, per-tenant** configuration flags Apache Fineract reads from each tenant database's `c_configuration` table. Unlike `application.properties` (which is read once at JVM start), these flags can be toggled at any time through the REST API and the change takes effect immediately for that tenant. They are the most direct way for administrators to enable or disable a Fineract feature without touching infrastructure.

**Source files**

- `fineract-core/src/main/java/org/apache/fineract/infrastructure/configuration/api/GlobalConfigurationConstants.java` — the canonical list of flag names.
- `fineract-provider/src/main/java/org/apache/fineract/infrastructure/configuration/api/GlobalConfigurationApiResource.java` — the REST API.
- `fineract-provider/src/main/java/org/apache/fineract/infrastructure/configuration/domain/ConfigurationDomainServiceJpa.java` — the runtime read service.

## The `c_configuration` table

Each tenant database contains a `c_configuration` table populated by Liquibase. Every row is a flag:

| Column | Type | Meaning |
| --- | --- | --- |
| `id` | bigint | Surrogate key. |
| `name` | varchar | Stable string identifier (e.g. `maker-checker`). Used by `GlobalConfigurationConstants`. |
| `enabled` | tinyint(1) | `true` enables the flag for this tenant. |
| `value` | bigint | Optional numeric value (e.g. days, retry count). |
| `string_value` | varchar | Optional string value. |
| `date_value` | date | Optional date value. |
| `description` | varchar | Free-text help shown in the admin UI. |
| `is_trap_door` | tinyint(1) | If `true`, the flag is one-way: once enabled it can never be disabled again. |

Trap-door flags are a deliberate safety measure for irreversible changes — for instance, switching the rounding mode after balances have accrued cannot be undone without a data migration.

## Why per-tenant flags exist

Apache Fineract serves financial institutions that disagree about almost everything: whether to require checker approval before a disbursement, whether to allow back-dated transactions, how often users should rotate passwords, whether to enable the address module or use the older client-fields workflow. Hard-coding any of these choices would force every operator into the same policy, while making them JVM-wide configuration would force every tenant in a multi-tenant Fineract deployment to share that policy.

The `c_configuration` table solves the problem by carrying the flag inside each tenant's own database. A single Fineract JVM can therefore serve "Bank A" with maker-checker on, "MFI B" with maker-checker off, and "Co-op C" with maker-checker on for loans but off for savings — all without restarting or reconfiguring the JVM.

## REST API

The full surface lives in `GlobalConfigurationApiResource` (`@Path("/v1/configurations")`):

```http
GET  /fineract-provider/api/v1/configurations              # list all flags (optional ?survey=true)
GET  /fineract-provider/api/v1/configurations/{configId}   # single flag by id
GET  /fineract-provider/api/v1/configurations/name/{name}  # single flag by name
PUT  /fineract-provider/api/v1/configurations/{configId}        # update by id
PUT  /fineract-provider/api/v1/configurations/name/{configName} # update by name
```

Update payload:

```json
{
  "enabled": true,
  "value": 90
}
```

All calls require the `CONFIGURATION` permission. Updates flow through Fineract's command pipeline, so they are auditable and (if `maker-checker` is on) they require a checker to approve before they take effect.

### Reading a flag from Java code

`ConfigurationDomainServiceJpa` is the read service; everywhere in Fineract that needs to consult a flag injects this bean and calls a typed accessor like `isMakerCheckerEnabledForTask(...)`, `retrievePenaltyWaitPeriod()`, or `isBusinessDateEnabled()`.

## Flags with values vs flags that are just booleans

Most of the flags above are pure booleans (the `enabled` column drives behaviour and the other columns are ignored). A meaningful subset carry an associated value:

- `force-password-reset-days` — `value` is the rotation period in days.
- `penalty-wait-period` — `value` is the wait period in days.
- `min-clients-in-group` / `max-clients-in-group` — `value` is the group-size bound.
- `daily-tpt-limit` — `value` is the daily third-party transfer limit.
- `financial-year-beginning-month` — `value` is `1`–`12`.
- `rounding-mode` — `value` is the ordinal of `java.math.RoundingMode`.
- `password-reuse-check-history-count` / `max-login-retry-attempts` — `value` is the count.

When you `PUT` an update for a value-carrying flag, supply both `enabled` and `value`:

```json
{ "enabled": true, "value": 90 }
```

For string-valued flags (`next-payment-due-date`, the asset-transfer strategy names, `report-export-s3-folder-name`) the column is `string_value` and the request payload uses `stringValue`. Date-valued flags (`organisation-start-date`) use `date_value` / `dateValue`. The API resource maps each variant for you.

## Reading flags in code

`ConfigurationDomainServiceJpa` is the central read-service every Fineract module consults to make a runtime decision. It caches the result of each `c_configuration` lookup using Fineract's cache abstraction (see `fineract.cache.default-template.*` in [Application Properties Reference](/configuration/application-properties#cache)), so the table is not hit on every business operation. The cache is invalidated when the corresponding flag is updated via the REST API.

A typical access pattern looks like:

```java
@Service
public class SomeBusinessService {
    private final ConfigurationDomainService configurationDomainService;

    public void doSomething(Loan loan) {
        if (this.configurationDomainService.isInterestChargedFromDateSameAsDisbursalDate()) {
            // use disbursal date
        } else {
            // use first-repayment date
        }
    }
}
```

The constant names in `GlobalConfigurationConstants` are the source of truth; if you are adding a new flag, you add a constant there and a Liquibase change-set that inserts the row into `c_configuration`.

## Survey configurations

`GET /v1/configurations?survey=true` returns the same table but filtered to the rows tagged as "survey" — used by the SPM (social performance management) module. The structure of each row is identical; only the curated subset differs.

## Difference between this and `application.properties`

A common source of confusion is the overlap between the layer-2 (`application.properties`) and layer-3 (`c_configuration`) flags. The rules of thumb:

- If a setting must be different across **environments** (dev/staging/prod) but the same across **tenants** in the same environment — it belongs in `application.properties`. Examples: database URL, S3 endpoint, JVM thread pool size.
- If a setting must be different across **tenants** that share the same JVM — it belongs in `c_configuration`. Examples: `maker-checker`, `enable-business-date`, `force-password-reset-days`.
- A handful of features have **both** an `application.properties` switch and a `c_configuration` flag (notably S3 for content and exports). In those cases the property is a "can this JVM use S3 at all" flag and the global configuration row is a "should this tenant use S3 in particular" flag — both must be on.

If you find yourself wanting to toggle the same setting per environment **and** per tenant, you have a design problem; pick one layer and live with it.

## Worked example — turning on maker-checker

```bash
# Find the flag
curl -k -u mifos:password \
  https://localhost:8443/fineract-provider/api/v1/configurations/name/maker-checker
```

```json
{
  "id": 4,
  "name": "maker-checker",
  "enabled": false,
  "trapDoor": false
}
```

```bash
# Enable it
curl -k -u mifos:password \
  -X PUT \
  -H 'Content-Type: application/json' \
  -d '{"enabled": true}' \
  https://localhost:8443/fineract-provider/api/v1/configurations/name/maker-checker
```

From the next request onwards, every command that is annotated as maker-checkable will be queued for checker approval instead of executing inline.

## Permissions model

The `CONFIGURATION` permission gates access to the API. In practice you should also wire it to maker-checker so that no single administrator can flip a flag without oversight. The standard pattern is:

1. Grant the `CONFIGURATION_READ` role to all staff who need to inspect flags.
2. Grant `CONFIGURATION_UPDATE` only to a small "admin" role.
3. Turn on the `maker-checker` flag itself so that future updates require a checker. Once on, the only way to disable maker-checker is through another maker-checker workflow — the system bootstraps its own oversight.

## Adding a new flag

If you are extending Fineract with a new feature flag, the steps are:

1. Add a constant to `GlobalConfigurationConstants.java` (in `fineract-core`).
2. Write a Liquibase change-set that inserts a row into `c_configuration` with the name, default `enabled` value, optional `value`/`string_value`/`date_value` and a description.
3. Inject `ConfigurationDomainService` (or `ConfigurationDomainServiceJpa`) into the service that consults the flag and add an accessor method (`isMyFeatureEnabled()`).
4. Use the accessor at runtime. The cache will populate on first call and invalidate on update.

Because the flag is per-tenant, the Liquibase change-set automatically runs against every tenant database, so all tenants get the new row.

## Notes & gotchas

<Warning>**Trap-door flags are irreversible.** Before enabling one (for instance changing `rounding-mode`), be sure you understand the data implications — there is no UI to disable it again.</Warning>

<Tip>Flags are **per-tenant**. If you run multi-tenant Fineract, you can have `maker-checker` on for one institution and off for another at the same time, on the same JVM.</Tip>

<Tip>For survey-related configuration use `GET /v1/configurations?survey=true` — the same endpoint returns a different curated subset.</Tip>
