Loan **state transitions** — approve, disburse, close — live on `LoansApiResource`. Everything that *posts money against* a loan or *attaches a charge to* a loan lives on a small family of sibling resources documented here. Together they cover: repayments, refunds, chargebacks, waivers, write-offs, schedule re-ages and re-amortisations, charge application and waiver, interest-pause variations, capitalized-income tracking for progressive loans, buy-down-fee amortization, and the post-dated cheque workflow used in several South-Asian deployments.

All endpoints mount under `/fineract-provider/api/v1` — see the [REST API Overview](/api/overview).

## `LoanTransactionsApiResource`

File: `fineract-provider/src/main/java/org/apache/fineract/portfolio/loanaccount/api/LoanTransactionsApiResource.java`
Class path: `@Path("/v1/loans")`

This is the single most overloaded resource in the platform. The POST handler accepts more than twenty distinct `?command=` values and routes each one to a separate `*CommandHandler`. The matrix below is the canonical reference:

| `?command=` | Handler | Produces |
| --- | --- | --- |
| `repayment` | `MakeRepaymentLoanCommandHandler` | `REPAYMENT` transaction. |
| `disburse` | `DisburseLoanCommandHandler` | `DISBURSEMENT` transaction (note: also reachable via `LoansApiResource` state transition). |
| `disburseToSavings` | `DisburseToSavingsCommandHandler` | `DISBURSEMENT` + linked savings deposit. |
| `merchantIssuedRefund` | `MerchantIssuedRefundCommandHandler` | `MERCHANT_ISSUED_REFUND`. |
| `payoutRefund` | `PayoutRefundCommandHandler` | `PAYOUT_REFUND`. |
| `goodwillCredit` | `GoodwillCreditCommandHandler` | `GOODWILL_CREDIT`. |
| `chargeRefund` | `ChargeRefundCommandHandler` | `CHARGE_REFUND`. |
| `chargeAdjustment` | `ChargeAdjustmentCommandHandler` | `CHARGE_ADJUSTMENT`. |
| `waiveinterest` | `WaiveInterestCommandHandler` | `WAIVE_INTEREST`. |
| `writeoff` | `WriteOffLoanCommandHandler` | `WRITEOFF`. |
| `close` | `CloseLoanCommandHandler` | `RECOVERY_REPAYMENT` + close. |
| `close-rescheduled` | `CloseLoanAsRescheduledCommandHandler` | Close with rescheduled flag. |
| `recoverypayment` | `RecoveryPaymentLoanCommandHandler` | Post-writeoff recovery. |
| `creditBalanceRefund` | `CreditBalanceRefundCommandHandler` | `CREDIT_BALANCE_REFUND`. |
| `refundByCash` | `RefundByCashLoanCommandHandler` | `REFUND_BY_CASH`. |
| `interestPaymentWaiver` | `InterestPaymentWaiverCommandHandler` | `INTEREST_PAYMENT_WAIVER`. |
| `down-payment` | `DownPaymentLoanCommandHandler` | `DOWN_PAYMENT`. |
| `accrual` | (manual periodic accrual) | `ACCRUAL`. |
| `reAge` | `ReAgeLoanCommandHandler` | Schedule re-age (no money). |
| `reAmortize` | `ReAmortizeLoanCommandHandler` | Schedule re-amortise (no money). |

The adjust handler (POST against an existing transaction id) accepts:

| `?command=` on `{transactionId}` | Effect |
| --- | --- |
| `undo` | Reverses the transaction. The reversal is itself written as a separate, audit-trail-preserving row with `is_reversed=true`. |
| `chargeback` | Splits the transaction into a chargeback. |
| `chargeAdjustment` | Apply a charge adjustment against a specific transaction. |

### Symmetry: external-id paths

Every POST and GET above has four URL shapes — `{loanId}`/`external-id/{loanExternalId}` × `{transactionId}`/`external-id/{transactionExternalId}` — and they all route to the same handler after id resolution. The class lists them in `LoansApiResourceSwagger.java`.

### Preview endpoints

`GET /v1/loans/{loanId}/transactions/reage-preview` and `/reamortization-preview` are read-only — they call the schedule generator with the same parameters that a `?command=reAge` / `reAmortize` POST would use, return the new schedule, and **do not** persist anything. The UI uses them so the credit officer can confirm before committing.

### Searching transactions

The resource also publishes the loan-transactions search endpoints — `GET /v1/loans/{loanId}/transactions/search` and `POST /v1/loans/{loanId}/transactions/query` (advanced JSON predicate). They are paginated and used heavily by the COB reconciliation UI.

## `LoanChargesApiResource`

File: `fineract-provider/src/main/java/org/apache/fineract/portfolio/loanaccount/api/LoanChargesApiResource.java`
Class path: `@Path("/v1/loans")`

This resource manages the link table between a loan and `m_charge`. A charge can be one of four time models: **disbursement**, **specified-due-date**, **installment-fee**, or **overdue-installment-fee**. The full lifecycle is:

1. `POST /v1/loans/{loanId}/charges` — apply (`m_loan_charge` row, not yet paid).
2. `POST /v1/loans/{loanId}/charges/{loanChargeId}?command=pay` — pay (writes `m_loan_transaction` of type `CHARGE_PAYMENT`).
3. `POST /v1/loans/{loanId}/charges/{loanChargeId}?command=waive` — waive.
4. `POST /v1/loans/{loanId}/charges/{loanChargeId}?command=adjustment` — adjust.
5. `PUT /v1/loans/{loanId}/charges/{loanChargeId}` — edit an unpaid amount.
6. `DELETE /v1/loans/{loanId}/charges/{loanChargeId}` — remove an unpaid charge.

The `template` endpoint returns the charges defined on the loan's product (the `charges` block of `LoanProductsApiResource`).

Every method has an external-id mirror — both for the loan and for the charge.

## `LoanInterestPauseApiResource`

File: `fineract-loan/src/main/java/org/apache/fineract/portfolio/interestpauses/api/LoanInterestPauseApiResource.java`
Class path: `@Path("/v1/loans")`

Interest pauses are time windows during which the loan does **not** accrue interest (used for grace periods, regulatory holds, public-health emergencies). They are stored in `m_loan_term_variations` with `term_type=INTEREST_PAUSE`.

| Method | Path | Handler |
| --- | --- | --- |
| GET | `/v1/loans/{loanId}/interest-pauses` | `retrieveInterestPauses` |
| POST | `/v1/loans/{loanId}/interest-pauses` | `createInterestPause` |
| PUT | `/v1/loans/{loanId}/interest-pauses/{variationId}` | `updateInterestPause` |
| DELETE | `/v1/loans/{loanId}/interest-pauses/{variationId}` | `deleteInterestPause` |
| GET | `/v1/loans/external-id/{loanExternalId}/interest-pauses` | `retrieveInterestPausesByExternalId` |
| POST | `/v1/loans/external-id/{loanExternalId}/interest-pauses` | `createInterestPauseByExternalId` |

Creating, editing or deleting a pause forces a schedule regeneration via `LoanScheduleService.regenerateRepaymentSchedule()`.

## `LoanCapitalizedIncomeApiResource`

| GET | `/v1/loans/external-id/{loanExternalId}/capitalized-incomes/external-id/{loanTransactionExternalId}` | `getCapitalizedIncomeAllocationDataByExternalIds` |

The matching write side (the capitalized-income transaction itself) is created during disbursement or via the standard `LoanTransactionsApiResource` flow with the appropriate command.

## `RepaymentWithPostDatedChecksApiResource`

File: `fineract-provider/src/main/java/org/apache/fineract/portfolio/repaymentwithpostdatedchecks/api/RepaymentWithPostDatedChecksApiResource.java`
Class path: `@Path("/v1/loans/{loanId}/postdatedchecks")`

Some markets accept a stack of post-dated cheques (PDCs) at disbursement time. The repayment job presents the appropriate cheque on its due date. This resource manages the cheque collection.

| Method | Path | Handler |
| --- | --- | --- |
| GET | `/v1/loans/{loanId}/postdatedchecks` | `getPostDatedChecks` |
| GET | `/v1/loans/{loanId}/postdatedchecks/{installmentId}` | `getPostDatedCheck` |
| PUT | `/v1/loans/{loanId}/postdatedchecks/{postDatedCheckId}` | `updatePostDatedChecks` |
| DELETE | `/v1/loans/{loanId}/postdatedchecks/{postDatedCheckId}` | `deletePostDatedCheck` |

The check creation endpoint lives on the parent loans resource (`POST /v1/loans/{loanId}?command=update`) — the cheques arrive in the disbursement payload alongside the schedule.

## Search and advanced query payloads

Both `GET /v1/loans/{loanId}/transactions/search` and `POST /v1/loans/{loanId}/transactions/query` accept the same filter primitives. The `search` form pushes them through query parameters (`?fromDate=…&toDate=…&transactionTypes=2,17,21`); the `query` form takes a JSON predicate:

```json
{
  "request": {
    "columnFilters": [
      { "column": "transaction_type_enum", "filters": [
          { "operator": "IN", "values": ["2", "17", "21"] }
      ]},
      { "column": "transaction_date", "filters": [
          { "operator": "GTE", "values": ["2024-01-01"] },
          { "operator": "LT",  "values": ["2024-04-01"] }
      ]}
    ],
    "resultsAreaColumnList": [ "id", "transaction_type_enum", "amount", "transaction_date" ],
    "paging": { "pageSize": 50, "pageNo": 0 },
    "sortOrder": [ { "column": "transaction_date", "direction": "DESC" } ]
  }
}
```

The same structure is supported on the savings counterpart.

## Worked example — repay and reconcile

```bash
TENANT='Fineract-Platform-TenantId: default'
HDR='Content-Type: application/json'

# Apply a one-off charge
curl -k -u admin:password -H "$TENANT" -H "$HDR" \
  -X POST https://localhost:8443/fineract-provider/api/v1/loans/911/charges \
  -d '{ "chargeId": 21, "amount": 5, "dueDate": "15 April 2024",
        "dateFormat": "dd MMMM yyyy", "locale": "en" }'

# Make a partial repayment
curl -k -u admin:password -H "$TENANT" -H "$HDR" \
  -X POST 'https://localhost:8443/fineract-provider/api/v1/loans/911/transactions?command=repayment' \
  -d '{ "transactionDate": "20 April 2024", "transactionAmount": 100, "paymentTypeId": 1,
        "dateFormat": "dd MMMM yyyy", "locale": "en" }'
# → { "loanId": 911, "resourceId": 8821 }

# Undo the repayment (the transaction id from above)
curl -k -u admin:password -H "$TENANT" -H "$HDR" \
  -X POST 'https://localhost:8443/fineract-provider/api/v1/loans/911/transactions/8821?command=undo' \
  -d '{ "transactionDate": "20 April 2024",
        "dateFormat": "dd MMMM yyyy", "locale": "en" }'
```

## Schema reference

- `m_loan_transaction` — every transaction row, joined to `m_loan` via `loan_id`.
- `m_loan_repayment_schedule` — the installment schedule that transactions consume.
- `m_loan_charge` — applied charges. `m_loan_charge_paid_by` links a charge to the repayment that paid it.
- `m_loan_term_variations` — interest pauses, payment holidays, principal variations.
- `m_loan_transaction_relation` — charge-back, refund and adjustment links between transactions.
- `m_loan_postdated_check` — post-dated cheques.
- `m_loan_transaction_repayment_period_mapping` — which installments a transaction touched (used by progressive loans).
- `m_loan_capitalized_income_allocation` and `m_loan_buy_down_fee_allocation` — progressive-loan amortisation tables.
