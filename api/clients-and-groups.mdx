The Apache Fineract API exposes the customer-master domain — clients, their identifying documents, charges and transactions, family members and addresses, as well as the group / centre aggregation that microfinance institutions use to organise lending — through a tight cluster of JAX-RS resources living in `fineract-provider/src/main/java/org/apache/fineract/portfolio/client/api/` and `…/portfolio/group/api/`. Every endpoint mounts under `/fineract-provider/api/v1` as described in the [REST API Overview](/api/overview).

This page lists the most important calls per resource and points at the source file for the long tail.

## `ClientsApiResource`

File: `fineract-provider/src/main/java/org/apache/fineract/portfolio/client/api/ClientsApiResource.java`
Class path: `@Path("/v1/clients")`

This is the workhorse of customer onboarding. Each method is a thin shell that either calls `clientReadPlatformService.retrieveOne(...)` or builds a `CommandWrapper` and hands it to `commandsSourceWritePlatformService.logCommandSource(...)`. The `@Path("{clientId}")` POST handler is a single dispatcher that looks at the `?command=` query parameter and translates it into the matching command name:

```java
// fineract-provider/.../portfolio/client/api/ClientsApiResource.java
@POST
@Path("{clientId}")
public String activate(@PathParam("clientId") final Long clientId,
        @QueryParam("command") final String commandParam,
        @Parameter(hidden = true) final String apiRequestBodyAsJson) {
    final CommandWrapper resourceDetails = builder.withClientId(clientId).withJson(apiRequestBodyAsJson)
            .activateClient(clientId).build();
    if (is(commandParam, "reject"))   { /* rejectClient(clientId) */ }
    if (is(commandParam, "withdraw")) { /* withdrawClient(clientId) */ }
    if (is(commandParam, "close"))    { /* closeClient(clientId) */ }
    // … reactivate, propose, assignStaff, unassignStaff, transfer, etc.
    return toApiJsonSerializer.serialize(commandsSourceWritePlatformService.logCommandSource(commandRequest));
}
```

That dispatch table is the single most useful reference for the client lifecycle — see `portfolio/clients.mdx` in this wiki for the full state diagram. The `external-id/{externalId}` overloads do the same things but use the tenant-supplied identifier instead of the numeric primary key.

Bulk endpoints `downloadtemplate` / `uploadtemplate` produce and consume the `.xls` workbook used by the bulk-import feature. They mount inside the same resource and delegate to `BulkImportApiResource` infrastructure.

## `ClientIdentifiersApiResource`

File: `fineract-provider/src/main/java/org/apache/fineract/portfolio/client/api/ClientIdentifiersApiResource.java`
Class path: `@Path("/v1/clients/{clientId}/identifiers")`

KYC identifier sub-resource. The `template` endpoint returns the configured identifier-type codes (e.g. National ID, Passport, Driving Licence) from the `m_code` / `m_code_value` tables. The standard CRUD set is:

| Method | Path | Handler | Notes |
| --- | --- | --- | --- |
| GET | `/v1/clients/{clientId}/identifiers/template` | `newClientIdentifierDetails` | Identifier-type code values. |
| GET | `/v1/clients/{clientId}/identifiers` | `retrieveAllClientIdentifiers` | All KYC documents for the client. |
| POST | `/v1/clients/{clientId}/identifiers` | `createClientIdentifier` | Add a new ID document. |
| GET | `/v1/clients/{clientId}/identifiers/{identifierId}` | `retrieveClientIdentifiers` | One identifier. |
| PUT | `/v1/clients/{clientId}/identifiers/{identifierId}` | `updateClientIdentifer` | Update. |
| DELETE | `/v1/clients/{clientId}/identifiers/{identifierId}` | `deleteClientIdentifier` | Delete. |

Uniqueness is enforced at the database layer (`uq_client_identifier`).

## `ClientChargesApiResource`

File: `fineract-provider/src/main/java/org/apache/fineract/portfolio/client/api/ClientChargesApiResource.java`
Class path: `@Path("/v1/clients/{clientId}/charges")`

Client-level fees (membership, annual, KYC processing, etc.). The `payOrWaiveClientCharge` handler is the multi-command POST — its dispatch table accepts `command=pay`, `command=waive`, `command=adjustment`. The price is taken from `m_charge` at the time the charge is applied; subsequent updates to the master charge do not propagate.

```java
@POST
@Path("{clientId}/charges/{chargeId}")
public String payOrWaiveClientCharge(
        @PathParam("clientId") final Long clientId,
        @PathParam("chargeId") final Long chargeId,
        @QueryParam("command") final String commandParam,
        @Parameter(hidden = true) final String apiRequestBodyAsJson) { … }
```

## `ClientTransactionsApiResource`

File: `fineract-provider/src/main/java/org/apache/fineract/portfolio/client/api/ClientTransactionsApiResource.java`
Class path: `@Path("/v1/clients")`

Read & undo of client-level transactions (the rows in `m_client_transaction`). Eight URL shapes exist because the resource is fully symmetrical between numeric id and `external-id` for both the client *and* the transaction:

| Path shape | Use |
| --- | --- |
| `/{clientId}/transactions` | Numeric id, list. |
| `/{clientId}/transactions/{transactionId}` | Numeric ids. |
| `/{clientId}/transactions/external-id/{transactionExternalId}` | Numeric client, external txn. |
| `/external-id/{clientExternalId}/transactions` | External client, list. |
| `/external-id/{clientExternalId}/transactions/{transactionId}` | External client, numeric txn. |
| `/external-id/{clientExternalId}/transactions/external-id/{transactionExternalId}` | Fully external. |

POSTs to any of those forms with no `?command=` are rejected; only `command=undo` is recognised — see the handler in `UndoClientTransactionCommandHandler`.

## `GroupsApiResource`

File: `fineract-provider/src/main/java/org/apache/fineract/portfolio/group/api/GroupsApiResource.java`
Class path: `@Path("/v1/groups")`

Groups in Fineract are *grouping_type=2* rows in `m_group` (centres are `grouping_type=1`). A group can own its own loans / savings (GLIM = Group Loan with Individual Monitoring, GSIM = Group Savings with Individual Monitoring) or simply be a container of clients.

Key endpoints:

| Method | Path | Handler | Purpose |
| --- | --- | --- | --- |
| GET | `/v1/groups/template` | `retrieveTemplate` | Code values, products, staff for the group-create UI. |
| GET | `/v1/groups` | `retrieveAll` | Paged search. |
| GET | `/v1/groups/{groupId}` | `retrieveOne` | `?associations=clientMembers,parentGroup,calendars,collectionMeetingCalendar,…`. |
| POST | `/v1/groups` | `create` | Pending or active group. |
| POST | `/v1/groups/{groupId}?command=…` | `activateOrGenerateCollectionSheet` | `activate`, `close`, `transferClients`, `associateClients`, `disassociateClients`, `assignRole`, `unassignRole`, `updateRole`, `generateCollectionSheet`, `saveCollectionSheet`. |
| POST | `/v1/groups/{groupId}/command/unassign_staff` | `unassignLoanOfficer` | Dedicated bulk-unassign call. |
| PUT | `/v1/groups/{groupId}` | `update` | Mutable fields. |
| DELETE | `/v1/groups/{groupId}` | `delete` | Pending only. |
| GET | `/v1/groups/{groupId}/accounts` | `retrieveAccounts` | All loans & savings for the group. |
| GET | `/v1/groups/{groupId}/glimaccounts` | `retrieveglimAccounts` | Group Loan Individual Monitoring children. |
| GET | `/v1/groups/{groupId}/gsimaccounts` | `retrieveGsimAccounts` | Group Savings Individual Monitoring children. |
| GET | `/v1/groups/downloadtemplate` | `getGroupsTemplate` | xls template for bulk import. |
| POST | `/v1/groups/uploadtemplate` | `postGroupTemplate` | xls bulk import. |

The collection-sheet endpoints feed into `CollectionSheetApiResource` — see `portfolio/collection-sheet.mdx`.

## `ClientSearchV2ApiResource`

File: `fineract-provider/src/main/java/org/apache/fineract/portfolio/client/api/v2/search/ClientSearchV2ApiResource.java`
Class path: `@Path("/v2/clients")`

The only **v2** endpoint in this domain. It hosts the new POST-body search that supersedes the older `/v1/clients?sqlSearch=…` query-parameter mechanism. It accepts a structured JSON predicate and returns a paged result with the same envelope as the v1 list endpoints.

| Method | Path | Handler |
| --- | --- | --- |
| POST | `/v2/clients/search` | `searchByText` |

Behind the scenes it goes through `SearchingClientRepository` in `fineract-core/.../portfolio/client/domain/search/`, which provides a JPA Criteria-based search that is friendlier to SQL injection auditing than the legacy `sqlSearch` parameter.

| `READ_FAMILYMEMBERS`, `CREATE_FAMILYMEMBERS`, `UPDATE_FAMILYMEMBERS`, `DELETE_FAMILYMEMBERS` | `ClientFamilyMembersApiResource`. |

## Common query parameters

The two big list endpoints (`/v1/clients` and `/v1/groups`) accept a shared set of paging and filtering parameters:

| Parameter | Purpose |
| --- | --- |
| `officeId` | Office-hierarchy scope — includes child offices. |
| `displayName`, `firstName`, `lastName` | Partial-match by name. |
| `externalId` | Exact-match by tenant id. |
| `status` | `all`, `pending`, `active`, `closed`, `rejected`, `withdrawn`, `transferInProgress`, `transferOnHold`. |
| `orphansOnly=true` | Clients not yet attached to a group (clients endpoint only). |
| `underHierarchy=…` | Hierarchy text from `m_office`. |
| `sqlSearch` | Legacy free-form predicate — prefer `ClientSearchV2ApiResource` for new code. |
| `paged=true`, `offset`, `limit`, `orderBy`, `sortOrder` | Pagination & sort. |

`?associations=` lets one read pull in linked data without a second round trip — values include `all`, `clientMembers` (groups), `parentGroup`, `charges`, `transactions`, `roles`, `meeting`, `calendars`, `collectionMeetingCalendar`, `parentHierarchy`.

## Worked example — onboard a client into a group

```bash
TENANT='Fineract-Platform-TenantId: default'
HDR='Content-Type: application/json'

# 1. Create the client (active immediately)
curl -k -u admin:password -H "$TENANT" -H "$HDR" \
  -X POST https://localhost:8443/fineract-provider/api/v1/clients \
  -d '{
    "officeId": 1, "firstname": "Jane", "lastname": "Doe",
    "externalId": "JD-0001",
    "active": true, "activationDate": "01 April 2024",
    "legalFormId": 1, "dateFormat": "dd MMMM yyyy", "locale": "en"
  }'
# → { "clientId": 4711, "resourceId": 4711 }

# 2. Attach the client to an existing group
curl -k -u admin:password -H "$TENANT" -H "$HDR" \
  -X POST "https://localhost:8443/fineract-provider/api/v1/groups/12?command=associateClients" \
  -d '{ "clientMembers": [4711] }'

# 3. Add a KYC identifier
curl -k -u admin:password -H "$TENANT" -H "$HDR" \
  -X POST https://localhost:8443/fineract-provider/api/v1/clients/4711/identifiers \
  -d '{ "documentTypeId": 1, "documentKey": "NID-12345678", "status": "Active" }'

# 4. Add an address (using the template's expected keys)
curl -k -u admin:password -H "$TENANT" -H "$HDR" \
  -X POST https://localhost:8443/fineract-provider/api/v1/client/4711/addresses \
  -d '{ "addressTypeId": 17, "city": "Nairobi", "countryIdCodeValue": 60,
        "isActive": true }'
```
