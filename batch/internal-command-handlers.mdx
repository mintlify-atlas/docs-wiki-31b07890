The Apache Fineract Batch API never reaches the database directly. Each
sub-request is matched against a regex routing table by
[`CommandStrategyProvider`](https://github.com/apache/fineract/blob/develop/fineract-core/src/main/java/org/apache/fineract/batch/command/CommandStrategyProvider.java),
which returns a Spring bean implementing
[`CommandStrategy`](https://github.com/apache/fineract/blob/develop/fineract-core/src/main/java/org/apache/fineract/batch/command/CommandStrategy.java).
The bean translates the `BatchRequest` into a regular method call on the
matching `*ApiResource` and wraps the response in a `BatchResponse`. This page
catalogues every concrete strategy that ships under
`fineract-provider/src/main/java/org/apache/fineract/batch/command/internal/`.

For background, see the [Batch API overview](/batch/overview). For the JAX-RS
resource that dispatches to the strategies, see
[BatchApiResource](/batch/batch-api-resource).

## The `CommandStrategy` SPI

Every handler implements one method from
[`CommandStrategy`](https://github.com/apache/fineract/blob/develop/fineract-core/src/main/java/org/apache/fineract/batch/command/CommandStrategy.java):

```java fineract-core/src/main/java/org/apache/fineract/batch/command/CommandStrategy.java
public interface CommandStrategy {

    BatchResponse execute(BatchRequest batchRequest, UriInfo uriInfo);
}
```

A canonical implementation just copies request fields onto a response, calls
the upstream `*ApiResource` bean, and sets a status code:

```java fineract-provider/src/main/java/org/apache/fineract/batch/command/internal/CreateClientCommandStrategy.java
@Component
@RequiredArgsConstructor
public class CreateClientCommandStrategy implements CommandStrategy {

    private final ClientsApiResource clientsApiResource;

    @Override
    public BatchResponse execute(final BatchRequest request, @SuppressWarnings("unused") UriInfo uriInfo) {

        final BatchResponse response = new BatchResponse();
        final String responseBody;

        response.setRequestId(request.getRequestId());
        response.setHeaders(request.getHeaders());

        responseBody = clientsApiResource.create(request.getBody());

        response.setStatusCode(HttpStatus.SC_OK);
        response.setBody(responseBody);

        return response;
    }
}
```

That gives every batched sub-request the same business logic, validation,
permissions and serialization as a direct REST call — the strategy is just a
thin adapter between the `BatchRequest`/`BatchResponse` envelope and the
existing resource.

## Routing rules

`CommandStrategyProvider.init()` maps a `(URL regex, HTTP method)` pair to a
Spring bean name. The regex toolkit is small:

```java fineract-core/src/main/java/org/apache/fineract/batch/command/CommandStrategyProvider.java
private static final String OPTIONAL_QUERY_PARAM_REGEX = "(\\?([\\w-]+=[^&]+)(?:&[\\w-]+=[^&]+)*)?";
private static final String MANDATORY_QUERY_PARAM_REGEX = "(\\?([\\w-]+=[^&]+)(?:&[\\w-]+=[^&]+)*)";
private static final String OPTIONAL_COMMAND_PARAM_REGEX = "(\\?command=[\\w\\-]+)?";
private static final String MANDATORY_COMMAND_PARAM_REGEX = "\\?command=[\\w\\-]+";
private static final String UUID_PARAM_REGEX = "[\\w\\d-]+";
private static final String NUMBER_REGEX = "\\d+";
private static final String ALPHANUMBERIC_WITH_UNDERSCORE_REGEX = "[a-zA-Z0-9_]*";
```

Older clients that send paths without the `v1/` prefix are auto-prefixed so
the routing regex sees a canonical URL:

```java fineract-core/src/main/java/org/apache/fineract/batch/command/CommandStrategyProvider.java
public CommandStrategy getCommandStrategy(final CommandContext commandContext) {
    if (isResourceVersioned(commandContext)) {
        return internalGetCommandStrategy(commandContext);
    } else {
        CommandContext alteredCommandContext = CommandContext.resource("v1/" + commandContext.getResource())
                .method(commandContext.getMethod()).build();
        return internalGetCommandStrategy(alteredCommandContext);
    }
}
```

If no regex matches, the provider returns a default
`UnknownCommandStrategy` that responds with a `501 Not Implemented` body so
the caller sees a clear error rather than a `NullPointerException`.

## Handlers grouped by domain

The 51 strategies under
`fineract-provider/src/main/java/org/apache/fineract/batch/command/internal/`
cover five domains (a 52nd, `UnknownCommandStrategy`, lives in
`fineract-core` as the fallback for unmatched routes). Each table lists:

- **Strategy** — the bean class name (file lives in the package above).
- **Method** — HTTP method for the sub-request.
- **`relativeUrl` pattern** — the regex from `CommandStrategyProvider.init`,
  rendered in human-friendly path form.
- **Upstream resource** — the `*ApiResource` the strategy forwards to.

### Account transfers

| Strategy | Method | `relativeUrl` pattern | Upstream resource |
| --- | --- | --- | --- |
| `CreateAccountTransferCommandStrategy` | POST | `v1/accounttransfers` | `PortfolioCommandSourceWritePlatformService.logCommandSource(...)` |

This handler is unusual: instead of forwarding to an `*ApiResource`, it
directly builds a `CommandWrapper` and pushes it through the command bus.
The upstream `AccountTransfersApiResource` does the same — the strategy just
inlines the construction so it can share the
`DefaultToApiJsonSerializer<CommandProcessingResult>` for the response body.

## Two handler patterns

Most strategies follow one of two implementation patterns.

<Tabs>
  <Tab title="Direct forward (simple)">
    The body and any path parameters are forwarded directly to the upstream
    resource. `CreateClientCommandStrategy` is the archetype: the
    `relativeUrl` has no path parameters because `POST /v1/clients` does not
    need one, so the strategy can call `clientsApiResource.create(...)` with
    only the body.

    ```java fineract-provider/src/main/java/org/apache/fineract/batch/command/internal/CreateClientCommandStrategy.java
    responseBody = clientsApiResource.create(request.getBody());
    response.setStatusCode(HttpStatus.SC_OK);
    response.setBody(responseBody);
    ```
  </Tab>
  <Tab title="Path-parameter splitter">
    When the URL embeds IDs (and possibly a `?command=` parameter), the
    strategy splits `relativeUrl` on `/` and `?` to extract them.
    `SavingsAccountTransactionCommandStrategy` is the archetype:

    ```java fineract-provider/src/main/java/org/apache/fineract/batch/command/internal/SavingsAccountTransactionCommandStrategy.java
    public BatchResponse execute(BatchRequest batchRequest, UriInfo uriInfo) {
        String relativeUrl = relativeUrlWithoutVersion(batchRequest);
        final List<String> pathParameters = Splitter.on('/').splitToList(relativeUrl);
        String command = null;
        // ...
    }
    ```

    `relativeUrlWithoutVersion` lives in
    [`CommandStrategyUtils`](https://github.com/apache/fineract/blob/develop/fineract-core/src/main/java/org/apache/fineract/batch/command/CommandStrategyUtils.java)
    and strips the optional `v1/` prefix so the same parser works for both
    versioned and unversioned URLs.
  </Tab>
</Tabs>

## What `UriInfo` is for

GET-style strategies need to forward query parameters to the upstream
resource. They wrap the inbound `UriInfo` in
`MutableUriInfo`, then call the same retrieval method an HTTP client would:

```java fineract-provider/src/main/java/org/apache/fineract/batch/command/internal/GetLoanByIdCommandStrategy.java
public BatchResponse execute(final BatchRequest request, final UriInfo uriInfo) {
    final MutableUriInfo parameterizedUriInfo = new MutableUriInfo(uriInfo);
    final BatchResponse response = new BatchResponse();
    // ... copy query params from relativeUrl into parameterizedUriInfo
    final String responseBody = loansApiResource.retrieveLoan(loanId, /* ... */, parameterizedUriInfo);
    response.setStatusCode(HttpStatus.SC_OK);
    response.setBody(responseBody);
    return response;
}
```

That is what lets a batched `GET v1/loans/42?associations=all` behave exactly
like a real HTTP `GET /v1/loans/42?associations=all`.

## Fallback: `UnknownCommandStrategy`

If no regex matches, `CommandStrategyProvider.internalGetCommandStrategy`
returns a default
[`UnknownCommandStrategy`](https://github.com/apache/fineract/blob/develop/fineract-core/src/main/java/org/apache/fineract/batch/command/internal/UnknownCommandStrategy.java)
instance. It produces a response indicating the URL or method is not
supported by the Batch API — important for early-failure feedback when a
client mistypes `relativeUrl` or sends a method that no strategy handles.

## Adding a new strategy

The process is mechanical:

<Steps>
  <Step title="Implement `CommandStrategy`">
    Create a new `@Component` under
    `fineract-provider/src/main/java/org/apache/fineract/batch/command/internal/`
    that injects the upstream `*ApiResource` bean and copies the
    `requestId`/`headers` onto the response, just like
    `CreateClientCommandStrategy`.
  </Step>
  <Step title="Register the route">
    Add a new `commandStrategies.put(...)` line in
    `CommandStrategyProvider.init()` with the URL regex, HTTP method and bean
    name (lower-camel-case of the class).
  </Step>
  <Step title="Write integration tests">
    Add a scenario to
    `integration-tests/src/test/java/org/apache/fineract/integrationtests/BatchRequestsIntegrationTest.java`
    that hits the new sub-request both standalone and chained through
    `reference`.
  </Step>
  <Step title="Document it">
    Add a row to the appropriate table on this page so the new operation is
    discoverable from the catalogue.
  </Step>
</Steps>
