Apache Fineract ships **~460 `*CommandHandler.java` classes** across its
Gradle modules. Every one of them is a Spring `@Service` that implements
`NewCommandSourceHandler` and carries a `@CommandType(entity, action)`
annotation. At startup, `CommandHandlerProvider` builds a single
`HashMap<String, String>` keyed by `entity + "|" + action` so that
`SynchronousCommandProcessingService` can hand a `CommandWrapper` to the
right handler in O(1). This page documents the discovery mechanism, the
handler package layout, and a representative sample of ~30 handlers grouped
by domain.

## The two-piece registration mechanism

### `@CommandType` — the marker

```java
// fineract-core/src/main/java/org/apache/fineract/commands/annotation/CommandType.java
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.TYPE)
@Documented
public @interface CommandType {

    /** Returns the name of the entity for this {@link CommandType}. */
    String entity();

    /** Return the name of the action for this {@link CommandType}. */
    String action();
}
```

The marker is intentionally minimal: two strings. Convention is uppercase
constants from
`fineract-core/.../commands/domain/CommandWrapperConstants.java`
(`ENTITY_LOAN`, `ACTION_DISBURSE`, …) but a handful of handlers reference
their own `*ApiConstants` (e.g.
`ClientApiConstants.CLIENT_CHARGES_RESOURCE_NAME`).

The combination `entity + "_" + action` is also the **task permission code**
(e.g. `DISBURSE_LOAN`), which is what `validateHasPermissionTo(...)` checks
inside `PortfolioCommandSourceWritePlatformServiceImpl.logCommandSource(...)`.

### `NewCommandSourceHandler` — the contract

```java
// fineract-core/src/main/java/org/apache/fineract/commands/handler/NewCommandSourceHandler.java
public interface NewCommandSourceHandler {

    CommandProcessingResult processCommand(JsonCommand command);
}
```

A single method. Every handler:

1. Pulls ids and parsed fields out of `JsonCommand`.
2. Delegates to a `*WritePlatformService` for the actual domain work.
3. Returns the service's `CommandProcessingResult` (or `CommandProcessingResult.empty()`).
4. Optionally wraps the call in `try/catch` for data-integrity issues and
   delegates to `DataIntegrityErrorHandler`.

### `CommandHandlerProvider` — bean scan + dispatch table

```java
// fineract-core/src/main/java/org/apache/fineract/commands/provider/CommandHandlerProvider.java
@Component
public class CommandHandlerProvider implements ApplicationContextAware, InitializingBean {

    private final HashMap<String, String> registeredHandlers = new HashMap<>();
    private ApplicationContext            applicationContext;

    @Override
    public void afterPropertiesSet() throws Exception {
        initializeHandlerRegistry();
    }

    private void initializeHandlerRegistry() {
        final String[] beans = applicationContext.getBeanNamesForAnnotation(CommandType.class);
        if (ArrayUtils.isNotEmpty(beans)) {
            for (final String name : beans) {
                final CommandType type = applicationContext.findAnnotationOnBean(name, CommandType.class);
                try {
                    if (type != null) {
                        registeredHandlers.put(type.entity() + "|" + type.action(), name);
                    } else {
                        log.error("Unable to register command handler '{}'!", name);
                    }
                } catch (final Throwable th) {
                    log.error("Unable to register command handler '{}'!", name, th);
                }
            }
        }
    }

    public NewCommandSourceHandler getHandler(final String entity, final String action) {
        Preconditions.checkArgument(StringUtils.isNoneEmpty(entity), "An entity must be given!");
        Preconditions.checkArgument(StringUtils.isNoneEmpty(action), "An action must be given!");

        final String key = entity + "|" + action;
        if (!registeredHandlers.containsKey(key)) {
            throw new UnsupportedCommandException(key);
        }
        return (NewCommandSourceHandler) applicationContext.getBean(registeredHandlers.get(key));
    }
}
```

Three behaviours worth knowing:

- **`applicationContext.getBeanNamesForAnnotation(CommandType.class)`** —
  Spring walks the bean factory, so any `@Service`/`@Component` annotated with
  `@CommandType` is picked up regardless of which Gradle module it lives in.
- **Map keyed by `entity|action`, valued by bean name** — the bean is fetched
  lazily on lookup, allowing prototype scopes (though all current handlers
  are singletons).
- **Duplicates overwrite silently** — if two handlers claim the same
  `entity|action`, the last one wins. The handler set is convention-driven, so
  this is in practice a non-issue.

`UnsupportedCommandException` is thrown for unknown keys; it surfaces as a
`501 Not Implemented` through Fineract's error mapper.

## Anatomy of a handler

The canonical pattern is two-method-thick:

```java
// fineract-loan/.../portfolio/loanaccount/handler/DisburseLoanCommandHandler.java
@Service
@RequiredArgsConstructor
@CommandType(entity = "LOAN", action = "DISBURSE")
public class DisburseLoanCommandHandler implements NewCommandSourceHandler {

    private final LoanWritePlatformService    writePlatformService;
    private final DataIntegrityErrorHandler   dataIntegrityErrorHandler;

    @Transactional
    @Override
    public CommandProcessingResult processCommand(final JsonCommand command) {
        try {
            return this.writePlatformService.disburseLoan(command.entityId(), command, false);
        } catch (final JpaSystemException | DataIntegrityViolationException dve) {
            dataIntegrityErrorHandler.handleDataIntegrityIssues(command, dve.getMostSpecificCause(), dve,
                    "loan.disbursement", "Disbursement");
            return CommandProcessingResult.empty();
        }
    }
}
```

And for a transactional repayment:

```java
// fineract-loan/.../portfolio/loanaccount/handler/LoanRepaymentCommandHandler.java
@Service
@RequiredArgsConstructor
@CommandType(entity = "LOAN", action = "REPAYMENT")
public class LoanRepaymentCommandHandler implements NewCommandSourceHandler {

    private final LoanWritePlatformService   writePlatformService;
    private final DataIntegrityErrorHandler  dataIntegrityErrorHandler;

    @Transactional
    @Override
    public CommandProcessingResult processCommand(final JsonCommand command) {
        try {
            boolean isRecoveryRepayment = false;
            return this.writePlatformService.makeLoanRepayment(LoanTransactionType.REPAYMENT,
                    command.getLoanId(), command, isRecoveryRepayment);
        } catch (final JpaSystemException | DataIntegrityViolationException dve) {
            dataIntegrityErrorHandler.handleDataIntegrityIssues(command, dve.getMostSpecificCause(), dve,
                    "loan.repayment", "Repayment");
            return CommandProcessingResult.empty();
        }
    }
}
```

Style notes:

- **`@Service` + `@RequiredArgsConstructor`** (Lombok) — every handler is a
  Spring singleton with constructor injection.
- **`@Transactional`** on the public method — handlers participate in the
  command bus's transaction (see
  [Command Implementation](/command/command-implementation)).
- **No fields beyond services** — handlers don't hold mutable state.
- **`DataIntegrityErrorHandler`** is the standard place to translate
  database constraint violations into platform exceptions.

## Special branches in `findCommandHandler`

Not every command goes through the `entity|action` map.
`SynchronousCommandProcessingService.findCommandHandler(wrapper)` has hand-coded
fast paths for two families of "transversal" operations:

```java
// fineract-core/.../SynchronousCommandProcessingService.java (excerpt)
if (wrapper.isDatatableResource()) {
    if (wrapper.isCreateDatatable()) {
        handler = applicationContext.getBean("createDatatableCommandHandler", NewCommandSourceHandler.class);
    } else if (wrapper.isDeleteDatatable()) {
        handler = applicationContext.getBean("deleteDatatableCommandHandler", NewCommandSourceHandler.class);
    } else if (wrapper.isUpdateDatatable()) {
        handler = applicationContext.getBean("updateDatatableCommandHandler", NewCommandSourceHandler.class);
    } else if (wrapper.isCreate()) {
        handler = applicationContext.getBean("createDatatableEntryCommandHandler", NewCommandSourceHandler.class);
    } else if (wrapper.isUpdateMultiple()) {
        handler = applicationContext.getBean("updateOneToManyDatatableEntryCommandHandler", NewCommandSourceHandler.class);
    } /* … */
} else if (wrapper.isNoteResource()) {
    if (wrapper.isCreate()) {
        handler = applicationContext.getBean("createNoteCommandHandler", NewCommandSourceHandler.class);
    } else if (wrapper.isUpdate()) {
        handler = applicationContext.getBean("updateNoteCommandHandler", NewCommandSourceHandler.class);
    } /* … */
}
```

- **Datatables** — handlers in
  `fineract-provider/.../infrastructure/dataqueries/handler/` are looked up by
  bean name because the URL pattern (`/datatables/...`) is the discriminator,
  not a fixed entity string.
- **Notes** — `fineract-provider/.../portfolio/note/handler/` is shared across
  client, group, loan, savings, and loan-transaction notes.

For everything else the `@CommandType` map wins.

## Worked example end to end

`POST /v1/loans/17/transactions?command=repayment` body `{"transactionAmount": 1000, …}`:

1. `LoansApiResource.executeCommand(...)` (in `fineract-provider/.../portfolio/loanaccount/api/`):
   ```java
   CommandWrapper commandRequest = new CommandWrapperBuilder()
           .loanRepaymentTransaction(loanId)
           .withJson(apiRequestBodyAsJson)
           .build();
   return this.commandsSourceWritePlatformService.logCommandSource(commandRequest);
   ```
2. `CommandWrapperBuilder.loanRepaymentTransaction(loanId)` sets
   `actionName = ACTION_REPAYMENT`, `entityName = ENTITY_LOAN`, `loanId = …`.
3. `PortfolioCommandSourceWritePlatformServiceImpl.logCommandSource` checks
   `validateHasPermissionTo("REPAYMENT_LOAN")`.
4. `SynchronousCommandProcessingService.executeCommand` saves the
   `UNDER_PROCESSING` `CommandSource`, then calls
   `CommandHandlerProvider.getHandler("LOAN", "REPAYMENT")`.
5. `CommandHandlerProvider` returns the bean named `loanRepaymentCommandHandler`.
6. `LoanRepaymentCommandHandler.processCommand(jsonCommand)` calls
   `loanWritePlatformService.makeLoanRepayment(LoanTransactionType.REPAYMENT, loanId, command, false)`.
7. The domain service returns a `CommandProcessingResult` that
   `SynchronousCommandProcessingService` then writes back as the row's `result`
   and flips status to `PROCESSED`.
