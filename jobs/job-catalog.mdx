This page is the authoritative list of every background job that Apache Fineract ships. Each row corresponds to one constant in:

```text fineract-core/src/main/java/org/apache/fineract/infrastructure/jobs/service/JobName.java
package org.apache.fineract.infrastructure.jobs.service;

public enum JobName {

    UPDATE_LOAN_ARREARS_AGEING("Update Loan Arrears Ageing"), //
    APPLY_ANNUAL_FEE_FOR_SAVINGS("Apply Annual Fee For Savings"), //
    ...
    WORKING_CAPITAL_LOAN_COB_JOB("Working Capital Loan COB"), //
    ;

    private final String name;
    JobName(final String name) { this.name = name; }
    @Override public String toString() { return this.name; }
}
```

The **enum-style name** (left of the parens) is the Spring Batch `Job` bean name registered with the `JobLocator`. The **display name** (the string in the constructor) is what gets stored in `job.name` and shown in the UI. The bridge:

```text fineract-provider/src/main/java/org/apache/fineract/infrastructure/jobs/config/JobNameProviderConfig.java
@Configuration
public class JobNameProviderConfig {

    @Bean
    public JobNameProvider fineractJobNameProvider() {
        List<JobNameData> jobNames = Arrays.stream(JobName.values())
            .map(jn -> new JobNameData(jn.name(), jn.toString())).toList();
        return new SimpleJobNameProvider(jobNames);
    }
}
```

A job that does not have a Spring Batch `@Bean Job` matching its enum-style name will not run â€” `JobRegisterServiceImpl.createJobDetail` throws `JobIsNotFoundOrNotEnabledException` and `JobSchedulerServiceImpl` logs and continues. That is intentional: it lets modules be packaged out of a deployment without breaking the scheduler.

## Concepts that recur in the table

A few terms are used in the **Effect / Trigger** column below:

- **Tasklet** â€” a single-step Spring Batch job whose `Step` is a `Tasklet`. Runs in one transaction, in one thread.
- **Chunk** â€” a Spring Batch chunk-oriented step (`reader â†’ processor â†’ writer`), still single-instance.
- **Partitioned** â€” the job has a `*Partitioner` and a manager step that hands worker steps off to either the same JVM (Spring Events transport) or other JVMs (JMS / Kafka transport). See `jobs/spring-batch-partitioned-jobs`.
- **Composite** â€” the job is configured in code as a sequence of multiple `Step` beans (e.g. `LOAN_COB` runs `resolveCustomJobParametersStep â†’ loanCOBStep â†’ stayedLockedStep`).

`JobName.LOAN_COB` is the canonical partitioned job. It is the only entry in the partitioned-job whitelist:

```text fineract-provider/src/main/java/org/apache/fineract/infrastructure/jobs/data/partitionedjobs/PartitionedJob.java
public enum PartitionedJob {
    LOAN_COB(LoanCOBConstant.LOAN_COB_PARTITIONER_STEP);

    @Getter private final String partitionerStepName;

    public static boolean existsByJobName(String jobName) {
        PartitionedJob partitionedJob = null;
        for (PartitionedJob job : values()) {
            if (jobName.equalsIgnoreCase(job.name())) {
                partitionedJob = job;
            }
        }
        return partitionedJob != null;
    }
}
```

`StuckJobExecutorServiceImpl` consults this enum to decide whether to use the partitioned-job restart path or the simple tasklet restart path (see `jobs/stuck-job-handling`).

`WORKING_CAPITAL_LOAN_COB_JOB` is functionally a partitioned job in the same family but is not currently in the `PartitionedJob` whitelist â€” its stuck-job recovery uses the tasklet path.

## Quick lookups

### By instance-mode gate

| Property | Jobs that go silent if `false` |
|----------|-------------------------------|
| `fineract.mode.batch-manager-enabled` | All 40 â€” `JobSchedulerServiceImpl` won't register them. |
| `fineract.mode.batch-worker-enabled` | `LOAN_COB` worker steps (partition handlers) â€” manager still schedules but workers can't pick up. |
| `fineract.job.loan-cob-enabled` | `LOAN_COB`, the inline `INLINE_LOAN_COB` and the LoanCOB API filter (`LoanCOBApiFilter`). |
| `fineract.job.journal-entry-aggregation.enabled` | `JOURNAL_ENTRY_AGGREGATION` |

### By trigger surface

| Trigger surface | Jobs / endpoints |
|-----------------|------------------|
| Cron via Quartz | All 40, when active and scheduled |
| `POST /v1/jobs/{id}?command=executeJob` | All 40, when batch-manager-enabled |
| `POST /v1/jobs/short-name/{shortName}?command=executeJob` | All 40 that have a `short_name` populated in the seed data (e.g. `SA_PINT` for `POST_INTEREST_FOR_SAVINGS`) |
| `POST /v1/jobs/{jobName}/inline` | `LOAN_COB`, `WC_LOAN_COB` only â€” gated by `InlineJobType` |
| Startup `ContextRefreshedEvent` catch-up | All 40 with `is_misfired = true` when `executeInstructionForMisfiredJobs = true` |
| Stuck-job restart at boot | All 40 whose `BATCH_JOB_EXECUTION.STATUS = STARTED` and below the retry threshold; partitioned path applies to `LOAN_COB` only |

### Implementation pattern shorthand

| Pattern | Examples |
|---------|----------|
| Single-step tasklet inside `jobs/service/<area>/<area>Config.java` | `UPDATE_NPA` (`updatenpa/UpdateNpaConfig.java` + `UpdateNpaTasklet.java`), `EXECUTE_DIRTY_JOBS` (`executealldirtyjobs/`), `INCREASE_BUSINESS_DATE_BY_1_DAY` (`increasedateby1day/increasebusinessdateby1day/`), `INCREASE_COB_DATE_BY_1_DAY` (`increasedateby1day/increasecobdateby1day/`) |

## How to find a job's bean

Given an enum-style name like `UPDATE_NPA`, the bean is registered as a Spring Batch `Job` whose **job name** (the first argument to `JobBuilder`) matches `JobName.UPDATE_NPA.name()`. For example:

```text fineract-provider/src/main/java/org/apache/fineract/infrastructure/jobs/service/updatenpa/UpdateNpaConfig.java
@Bean
public Job updateNpaJob() {
    return new JobBuilder(JobName.UPDATE_NPA.name(), jobRepository).start(updateNpaStep()).incrementer(new RunIdIncrementer()).build();
}
```

The `JobLocator` (provided by Spring Batch) discovers these by their job name (`JobName.UPDATE_NPA.name()` â†’ `"UPDATE_NPA"`). That is the name `jobLocator.getJob(...)` is called with in `JobRegisterServiceImpl.createJobDetail`.

A handy way to find any job's wiring is:

```bash
grep -rln "JobName\.<ENUM_NAME>\.name()" fineract-provider/src/main/java
```

## What is NOT in `JobName`

The `JobName` enum is the catalog of **Spring Batch jobs**. A few things look like jobs but are not:

- The **`COBBusinessStep`** chain â€” these are sub-units of `LOAN_COB`, not free-standing jobs. See `cob/business-steps`.
- The **report-mailing definitions** â€” these are stored in `report_mailing_job` and are *fired* by the `EXECUTE_REPORT_MAILING_JOBS` job (#21), not as separate `JobName` entries.
- The **standing instructions** â€” same pattern as above; `EXECUTE_STANDING_INSTRUCTIONS` (#9) drives them.
- The **scheduler control** itself (`/v1/scheduler` `start` / `stop`) â€” that's a state flip on `scheduler_detail`, not a job.
- **Inline COB execution** â€” that uses `InlineJobType` (an enum with two values: `LOAN_COB` and `WC_LOAN_COB`), which maps **inline** job tokens (`LOAN_COB`, `WC_LOAN_COB`) to **inline executor classes** (`InlineLoanCOBExecutorServiceImpl`, `InlineWorkingCapitalLoanCOBExecutorServiceImpl`). The Spring Batch job those executors invoke under the hood is named `INLINE_LOAN_COB` / `INLINE_WORKING_CAPITAL_LOAN_COB`, not in `JobName`. See `jobs/inline-job-execution`.

## How the catalog is exposed to the SPI

Other Gradle modules can contribute additional jobs without touching `JobName` by registering their own `JobNameProvider` bean:

```text fineract-provider/src/main/java/org/apache/fineract/infrastructure/jobs/service/jobname/JobNameProvider.java
public interface JobNameProvider {
    Set<JobNameData> provide();
}
```

```text fineract-provider/src/main/java/org/apache/fineract/infrastructure/jobs/service/jobname/SimpleJobNameProvider.java
public class SimpleJobNameProvider implements JobNameProvider {

    private final Set<JobNameData> jobNames;

    public SimpleJobNameProvider(Collection<JobNameData> jobNames) {
        this.jobNames = Set.copyOf(jobNames);
    }

    @Override
    public Set<JobNameData> provide() {
        return jobNames;
    }
}
```

`JobNameService` flattens every registered provider's set and is what `JobRegisterServiceImpl` consults to translate the display name from `job.name` back to the enum-style key.

```text fineract-provider/src/main/java/org/apache/fineract/infrastructure/jobs/service/jobname/JobNameService.java
@Component
@RequiredArgsConstructor
public class JobNameService {

    private final Collection<JobNameProvider> providers;

    public JobNameData getJobByHumanReadableName(String jobName) {
        Optional<JobNameData> optionalJob = getJobNames().stream()
            .filter(jn -> jobName.equals(jn.getHumanReadableName())).findAny();
        return optionalJob.orElseThrow(() -> new IllegalArgumentException("Job not found by name: " + jobName));
    }

    private Set<JobNameData> getJobNames() {
        return providers.stream().map(JobNameProvider::provide).flatMap(Collection::stream).collect(toSet());
    }
}
```

If you add a job in a downstream module, ship a `JobNameProvider` bean with it and Fineract's scheduler will discover it.

## The `JobName` lifecycle

A new constant in `JobName.java` is **not enough** to make a job runnable. The end-to-end checklist for adding a job:

1. **Add the enum constant** to `fineract-core/src/main/java/org/apache/fineract/infrastructure/jobs/service/JobName.java`. Pick a stable display name â€” it is what operators will see and what the `job` table will store.

2. **Register a Spring Batch `@Bean Job`** whose **Spring Batch job name** (the first argument to `JobBuilder`) matches `JobName.MY_NEW_JOB.name()`. The bean can live in a `@Configuration` that is `@ConditionalOnProperty` or `@Conditional` if it needs to be disable-able with a flag. Example shape:

   ```java
   @Configuration
   public class MyNewJobConfig {
       @Bean
       public Job myNewJob(JobRepository jobRepository, ...) {
           return new JobBuilder(JobName.MY_NEW_JOB.name(), jobRepository)
                   .start(myStep())
                   .incrementer(new RunIdIncrementer())
                   .build();
       }
   }
   ```

3. **Add a row to the `job` table** via a Liquibase changelog so the scheduler can see it on first boot. The row needs at minimum: `name` (display name), `display_name`, `cron_expression`, `is_active`, `task_priority`, `group_name`, `node_id`, `short_name`, `updates_allowed`. See changelogs under `fineract-provider/src/main/resources/db/changelog/tenant/parts/` for prior examples.

4. **(Optional) Add to `PartitionedJob`** if the job uses a manager / worker partitioned step. Required for `StuckJobExecutorServiceImpl` to use the partitioned restart path. See `jobs/spring-batch-partitioned-jobs`.

5. **(Optional) Add tuning properties** to `application.properties` under `fineract.partitioned-job.partitioned-job-properties[i]` if partitioned, or a dedicated `fineract.job.<my-job>.*` namespace if you want runtime knobs.

6. **(Optional) Add command-source permissions** in the `m_permission` seed data if you want users to be able to fire it via `POST /v1/jobs/{id}?command=executeJob`. The default permission is `EXECUTEJOB_SCHEDULER`.

7. **(Optional) Add to `InlineJobType`** only if your job is loan-scoped and you have an `InlineExecutorService` implementation for it. See `jobs/inline-job-execution`.

A job that omits step 2 (no Spring Batch bean) will show up in the `/v1/jobs` list but every fire will throw `JobIsNotFoundOrNotEnabledException` and a warning will be logged at boot:

```text fineract-provider/src/main/java/org/apache/fineract/infrastructure/jobs/service/JobSchedulerServiceImpl.java
try {
    jobRegisterService.scheduleJob(jobDetails);
} catch (JobIsNotFoundOrNotEnabledException e) {
    log.warn("{}", e.getMessage());
}
```

This is intentional: a job that's been disabled in this deployment should not crash startup.

## Reading the run history

For any job in the catalog, the per-fire history is available at:

```text
GET /v1/jobs/{jobId}/runhistory?offset=0&limit=200
GET /v1/jobs/short-name/{shortName}/runhistory?offset=0&limit=200
```

Each row corresponds to one Quartz fire (cron or run-now), with fields:

| Field | From | Meaning |
|-------|------|---------|
| `version` | `ScheduledJobRunHistory.version` | Monotonically increasing per job |
| `jobRunStartTime` | `context.getFireTime()` | When Quartz invoked the job |
| `jobRunEndTime` | end of `SchedulerJobListener.jobWasExecuted` | When Quartz finished |
| `status` | `success` / `failed` | `SchedulerServiceConstants.STATUS_*` |
| `triggerType` | `cron` / `application` | What kicked it off |
| `jobRunErrorMessage` | extracted by `getCauseFromException` | The root-cause message |
| `jobRunErrorLog` | full stack trace | The full unwound stack |

Failure rows are written by `SchedulerJobListener` (see `jobs/scheduler-and-quartz`). Inline runs **do not** produce these rows â€” they go to `BATCH_JOB_EXECUTION` only.
